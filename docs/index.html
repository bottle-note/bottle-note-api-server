1. 개요 (overview)

  • 이 API문서는 bottle-note 프로젝트의 API를 설명합니다.

  • 포함되면 좋을것 같은 내용이나 수정이 필요한 내용은 언제든지 디스코드 채널을 통해 알려주세요.

1.4. 공통 인증/인가

보틀노트 서비스에서는 인증이 필요한 URL에 접근하기 위해서는 Authorization 헤더에 로그인 시 발급받은 Access Token을 첨부해서 요청을 보내야 합니다.

아래 예시는 사용자가 로그인을 하지 않은 경우나 Authorization 헤더에 첨부한 토큰에 따라 발생할 수 있는 예외의 예시입니다.

  • Authorization 헤더에 액세스 토큰을 첨부하지 않으면 토큰이 존재하지 않습니다 에러 메시지가 출력됩니다 (400)

  • Authorization 헤더에 첨부한 토큰이 잘못 된 토큰이면 잘못된 토큰입니다 라는 에러 메시지가 출력됩니다 (401)

  • Authorizatioin 헤더에 첨부한 토큰이 만료 된 토큰이면 만료된 토큰입니다 라는 에러 메시지가 출력됩니다 (403)

2. 공통 (common) API

2.1. 이미지 업로드 경로 요청

이미지를 업로드 하기 위한 업로드 경로를 요청합니다.

이미지 업로드는 인증된 유저만 요청 가능합니다.

업로드 시에는 S3를 대상으로 진행합니다.

조회 시에는 CDN 서비스를 통해 이미지를 제공합니다.

요청 파라미터

GET /api/v1/s3/presign-url?rootPath=images&uploadSize=1 HTTP/1.1
Host: localhost:8080
Parameter Description

rootPath

업로드 파일 경로 (하단 설명 참조)

uploadSize

업로드할 이미지의 사이즈 ( 이미지당 1개 )

  • rootPath : 루트 경로로 부터 이미지 저장 경로를 지정합니다

    • rootPath 가 review 일 경우 review/ 로 시작하는 경로로 이미지가 저장됩니다.

      • rootPath 가 review 일 경우 서버 에서는 해당 업로드일과 랜덤한 UUID를 할당한 경로를 생성합니다.

        • ex. review/20210101/random-uuid.jpg / user/profile/20210101/random-uuid.jpg

      • 이때 이미지의 원본명은 필요 없습니다.(현재로서는)

    • rootPath는 하나의 디렉토리라고 생각하고 일관된 상수로 프론트 영역에서 관리합니다 .

      • ex) review, user/profile,user/album, product 등

    • 문자열 rootPath의 시작과 끝의 /은 무시 됩니다.

  • 최소 한 디렉토리 이상의 rootPath 를 지정해야 합니다.

  • uploadSize는 업로드할 사진의 크기만큼 요청합니다.

    • uploadSize는 최대 10장 까지 가능합니다.

  • uploadSize 파라미터를 전달하지 않을 경우 1장이 기본값으로 설정됩니다.

3.3. 닉네임 변경

닉네임 변경 API는 사용자가 로그인 후 자신의 닉네임을 변경할 수 있게 해줍니다.

닉네임 변경 요청은 다음과 같은 조건을 만족해야 합니다:

  • 닉네임은 중복될 수 없습니다.

  • 닉네임은 최소 2자 이상, 최대 11자 이하여야 합니다.

  • 닉네임은 한글, 영문, 숫자만 사용할 수 있습니다.

  • 닉네임은 공백을 포함할 수 없습니다.

  • userId와 newNickname은 필수 요청 필드입니다.

6. 리뷰 댓글 (review reply) 관련 API

6.1. 리뷰 댓글 등록

리뷰의 댓글을 등록 하는 API 문서입니다.

PathVariable로 댓글이 작성되는 대상 리뷰의 식별자를 전달합니다.

  • 댓글의 상위는 크게 두가지로 구분됩니다

    • root reply id : 최상위 댓글 식별자

    • parent reply id : 상위 댓글 식별자

최상위 식별자의 경우 서버에서 파악하기 때문에 상위 댓글 식별자만 전달이 필요합니다.

예를 들어

├── 첫번째 댓글
└───├── 두번째 댓글
    └────├─ 세번째 댓글
         └──── 네번째 댓글

  • 첫번쨰 댓글은 상위/최상위 부모가 없는 댓글입니다.

  • 두번째 댓글의 상위 부모이자 최상위 댓글은 첫번째 댓글입니다.

  • 세번째 댓글의 상위 부모는 두번째 댓글이고 최상위 댓글 식별자는 첫번째 댓글의 식별자입니다.

  • 네번째 댓글의 상위 부모는 세번째 댓글이고 최상위 댓글 식별자는 첫번째 댓글의 식별자입니다.

  • 계층 구조는 이해를 쉽게 하기 위한 예시이며 실제 조회시에는 최대 1Depth로 조회됩니다.

6.3. 하위 리뷰 댓글 목록 조회

리뷰의 대댓글 목록을 조회하는 API입니다.

경로의 rootReplyId 는 최상위 댓글 목록 조회 API 통해 조회한 댓글의 식별자를 전달합니다.

구조 예 ).

  • 최상위 리뷰 댓글 목록 조회 API 를 통해 Root Reply을 조회합니다.

  • 하위 리뷰 댓글 목록 조회 API 를 통해 Root Reply에 대한 대댓글을 조회합니다.

  • 2번 댓글의 하위 댓글을 조회 하기 위해 2번 댓글의 식별자를 pathVariable 로 전달합니다

  • 댓글의 관계는 최상위 댓글::하위 댓글 과 부모 댓글: 자식 댓글 관계로 구분됩니다.

    • 최상위 댓글::하위 댓글 :1:N의 관계를 가집니다.
      최상위 댓글만 대상이 될수 있습니다.

    • 부모 댓글: 자식 댓글 : 1:N의 관계를 가집니다.
      부모 댓글은 최상위 댓글이 아니여도 대상이 될수 있습니다.

      • 2-1번 댓글의 상위 댓글과 최상위 댓글은 모두 2번 댓글입니다.

      • 2-2번 댓글의 상위 댓글은 2-1번 댓글이며 최상위 댓글은 2번 댓글입니다.

      • 2-3번 댓글의 상위 댓글은 2-1번 댓글이며 최상위 댓글은 2번 댓글입니다.

      • 2-4번 댓글의 상위 댓글은 2-3번 댓글이며 최상위 댓글은 2번 댓글입니다.

  • 서비스의 모든 정렬 관련 API에서 공통적으로 사용됩니다.

DESC

내림차순

5 → 3 → 2 → 1

ASC

오름차순

1 → 2 → 3 → 5

  • 랜덤의 경우 정렬 타입에 따라 의도와 다르게 동작 할 수 있습니다.

  • 해당 값이 없을 경우 기본값은 DESC(내림차순) 입니다.

  • 타입이 잘못 전달될 경우 예외가 반환됩니다.

# Project Conventions

> Generated by /scan-conventions on 2026-05-14. Re-run if the project structure changes significantly. Partially updated on 2026-06-12 to reflect the bottlenote-test-support module split (PR #623).

## Detected Project Type

Hybrid Spring Boot multi-module backend.

- `bottlenote-product-api`: public web API, Java, executable `bootJar`
- `bottlenote-admin-api`: admin web API, Kotlin, executable `bootJar`
- `bottlenote-mono`: core/domain module, Java, library JAR
- `bottlenote-test-support`: shared test infrastructure and fixtures (TestContainers config, TestFactory, InMemory repositories), Java, library JAR consumed via `testImplementation`
- `bottlenote-batch`: Spring Batch + Quartz module, Java, executable `bootJar`
- `bottlenote-observability`: observability library module, Java

The current batch-focused work must treat `bottlenote-batch` as the primary surface and `bottlenote-mono`, `git.environment-variables`, and `bottlenote-observability` as likely impact neighbors.

## Build System & Language

- Language: Java 21 across Java modules; Kotlin 1.9.23 in `bottlenote-admin-api`
- Framework: Spring Boot 3.4.11, Spring Cloud 2024.0.1
- Build: Gradle multi-module with version catalog at `gradle/libs.versions.toml`
- Packaging: root disables `bootJar` by default for subprojects, then app modules re-enable it
- Formatter: Spotless with `googleJavaFormat("1.27.0")` for Java modules and ktlint for admin Kotlin
- Test runner: Gradle `Test` with JUnit Platform tag-filtered tasks
- Documentation: Spring REST Docs + Asciidoctor in product/admin modules
- QueryDSL: configured for `bottlenote-mono` and `bottlenote-product-api`

## Naming Conventions

- Java application entrypoint: `{Product|Batch}Application.java`
- Kotlin admin entrypoint: `Application.kt` with `AdminApplication`
- Product controllers: `{Domain}{Action}Controller.java` under `controller`
- Admin controllers: `Admin{Domain}Controller.kt` under `presentation`
- Core services: `{Domain}Service.java`, or existing command/query variants such as `AdminAlcoholCommandService`
- Facade interface: `{Domain}Facade.java` under `facade`
- Facade implementation: current code commonly uses `Default{Domain}Facade.java` under `service`
- Domain repository interface: `{Domain}Repository.java` under `domain`
- JPA repository implementation: `Jpa{Domain}Repository.java` under `repository`
- QueryDSL custom types: `Custom{Domain}Repository.java`, `Custom{Domain}RepositoryImpl.java`, `{Domain}QuerySupporter.java`
- Exception model: `{Domain}Exception.java` and `{Domain}ExceptionCode.java`
- Batch job config: `{JobName}JobConfig.java`
- Batch Quartz binding: nested `{JobName}QuartzJob extends BatchQuartzJob`
- Test class examples: `{Domain}ServiceTest`, `{Domain}IntegrationTest`, `Rest{Domain}ControllerDocsTest`, `Admin{Domain}ControllerDocsTest`
- Display names: Korean `@DisplayName`, often describing observable behavior

## Test Patterns

- Categorization:
  - `@Tag("unit")`: unit tests, run by `./gradlew unit_test`
  - `@Tag("integration")`: product integration tests, run by `./gradlew integration_test`
  - `@Tag("admin_integration")`: admin integration tests, run by `./gradlew admin_integration_test`
  - `@Tag("rule")`: ArchUnit rules, run by `./gradlew check_rule_test`
  - `@Tag("batch")`: batch tests, run by `./gradlew batch_test`; no batch test source files were present during scan
- Root `test` excludes `data-jpa-test`, `integration`, and `admin_integration`; it does not currently exclude RestDocs.
- Product integration base: `bottlenote-product-api/src/test/java/app/bottlenote/IntegrationTestSupport.java`
- Admin integration base: `bottlenote-admin-api/src/test/kotlin/app/IntegrationTestSupport.kt`
- Testcontainers config: `bottlenote-test-support/src/main/java/app/bottlenote/operation/utils/TestContainersConfig.java`
- Fake/InMemory pattern is widely present under test `fixture/` or `fake/` packages.
- Mockito exists and is used in some unit and docs tests, but project guidance is Fake/InMemory first.
- Known test drift observed during scan:
  - `@Tag("batch")` count was 0.
  - Some tests have commented-out `@Test`.
  - At least one `@Disabled` test remains.
  - `bottlenote-batch` has a context wiring test (BatchApplicationContextTest) using Testcontainers MySQL.

## Architecture Markers

- Layer markers:
  - `@DomainRepository`: pure domain repository marker
  - `@JpaRepositoryImpl`: JPA implementation marker, includes Spring `@Repository`
  - `@FacadeService`: facade implementation marker, includes Spring `@Service`
  - `@DomainEventListener`: domain event listener marker
  - `@ThirdPartyService`: external integration marker
  - `@SecurityPolicy`, `@ExcludeRule`: security/rule-related project markers
- Web API layering:
  - Controller extracts request/security context and delegates to service/facade.
  - Service owns transaction boundary.
  - Repository interfaces generally live in `domain`; JPA/QueryDSL implementation lives in `repository`.
- Response model:
  - `GlobalResponse` shape is `success`, `code`, `data`, `errors`, `meta`.
  - `GlobalExceptionHandler` maps `AbstractCustomException`, validation, JWT, JSON, AWS, and generic exceptions.
- Persistence:
  - JPA, QueryDSL, MySQL, Redis.
  - SQL resources for batch jobs are loaded from classpath paths under `storage/mysql/sql`.
- Batch architecture:
  - `BatchApplication` uses broad `scanBasePackages = "app"` and `@ComponentScan(basePackages = "app")`.
  - `BatchQuartzJob` is the common Quartz-to-Spring-Batch bridge.
  - Job config classes define `Job`, `Step`, readers/writers, payload records, and nested Quartz job bindings.
  - Batch jobs use `JdbcTemplate`, `JobRepository`, `PlatformTransactionManager`, and SQL resources from classpath.
- Observability:
  - Separate `bottlenote-observability` module provides AOP logging/tracing and actuator/OTel/Micrometer dependencies.

## Batch-Specific Current Conventions

- Main class sets timezone to `Asia/Seoul`.
- `spring.batch.job.enabled=false`; jobs are launched through Quartz bindings rather than auto-running on startup.
- Quartz uses JDBC job store in main config and memory job store in batch test resource config.
- Batch app exposes health only through actuator.
- Batch datasource pool is intentionally small in `application-datasource.yml`.
- Shared test fixtures/infra live in `bottlenote-test-support` (main sourceSet); mono/product/admin consume it via `testImplementation`. The former mono `sourceSets.test.output` hack was removed in PR #623; batch needs no test-support dependency.
- Batch currently includes `git.environment-variables` in both main and test resources.
- Batch currently contains hardcoded JWT secret and nonce salt defaults in main resource config.
